## 一、目标与范围

本次需要一个“像 json.cn 那样”的 JSON 解析器功能，用于将用户输入的一串 JSON 解析、格式化，并以更易读、更直观的方式呈现。

**核心目标**

1. 让用户把混乱、压缩、难读的 JSON 快速变成结构清晰、一眼能看懂的格式。
2. 在博客的“工具箱”中提供一个轻量、顺手、稳定的小工具。
3. 保持纯前端实现，不上传用户输入，不引入后端依赖。

**范围**

本次只做 JSON 解析与可视化相关能力，不做：

- JSONPath 查询
- JSON 转 CSV / Markdown
- 两个 JSON 对比
- 登录态、云端历史记录、协作分享

## 二、页面入口与信息架构

### 2.1 入口

工具放在工具箱模块中，作为独立子页面存在：

+ 入口为一个卡片，名为JSON解析器，带一个图标，和一句话的简介

- 路由：`/toolbox/json-parser`
- 导航路径建议：首页 / 工具箱入口 -> JSON 解析器

### 2.2 页面结构

页面采用左右分栏布局，结构如下：

1. **页面头部**
   - 标题：`JSON 解析器`
   - 副标题或简短说明：提示“粘贴 JSON 即可格式化与查看”

2. **左侧输入区**
   - JSON 输入区
   - 主要操作按钮：格式化、压缩、清空

3. **右侧结果区**
   - 结果视图切换：`文本 / 结构`
   - 次要操作：复制结果、下载结果
   - 错误提示区：解析失败时展示错误信息

## 三、目标用户与使用场景

### 3.1 目标用户

- 博客读者 / 开发者 / 学生 / 内容创作者
- 需要快速看懂一段 JSON 的人
- 不想每次都打开重型 IDE 或专业工具的人

### 3.2 典型场景

1. 复制了一段压缩 JSON，想快速展开阅读。
2. 调试接口返回值，想检查层级结构。
3. 学习 JSON 结构，通过格式化结果理解层级。
4. 粘贴文本后遇到语法错误，想快速定位问题。

## 四、功能需求

### 4.1 核心功能

#### F1. JSON 输入

- 支持在输入区粘贴任意字符串。
- 输入为空时，右侧展示占位提示，不触发报错。
- 输入变化后应自动触发解析与更新，但需做节流，避免高频输入时频繁抖动。

#### F2. 格式化 JSON

- 将输入字符串解析为 JSON，并以缩进形式重排输出。
- 输出应保持原始结构，不修改数据内容。
- 格式化后的结果应便于肉眼阅读层级。

#### F3. 压缩 JSON

- 将输入 JSON 压缩为单行形式。
- 适合复制到接口请求、配置文件等空间有限场景。

#### F4. 双视图切换

右侧结果区支持两种视图：

- **文本视图**
  - 展示格式化后的 JSON 文本。
  - 需要带行号。
  - 需要做 JSON 语法高亮。
  - 默认展示该视图。

- **结构视图**
  - 将 JSON 以树形结构展示。
  - 对象和数组支持折叠 / 展开。
  - 键值显示应清晰、层级可辨。
  - 适合复杂嵌套 JSON 的快速浏览。

#### F5. 复制

- 支持复制左侧输入。
- 支持复制右侧结果。
- 复制成功后需要有轻量提示，例如按钮文案变化或 toast。

#### F6. 下载结果

- 支持将右侧格式化结果下载为 `.json` 文件。
- 下载内容默认为格式化文本，而不是结构视图的序列化产物。

#### F7. 清空输入

- 一键清空左侧输入。
- 清空后右侧回到默认占位状态，不再显示旧结果。

#### F8. 错误提示

当输入不是合法 JSON 时，需要给出明确提示：

- 提示文案应通俗易懂，避免只展示技术栈错误信息。
- 如果能解析出错误位置，应展示行号或字符位置。
- 推荐格式示例：
  - `Line 12, Column 5: Unexpected token }`
- 错误提示应靠近输入区或结果区顶部，容易被第一时间看到。

## 五、交互设计

### 5.1 默认状态

- 左侧为空输入状态，显示占位提示，例如：`在此粘贴 JSON ...`
- 右侧默认展示文本视图。
- 操作按钮默认可点击，但在无意义操作状态下应禁用，例如：
  - 输入为空时，`复制输入` / `下载结果` 可禁用或给出友好提示。

### 5.2 输入反馈

- 用户输入变化后，系统自动尝试解析。
- 如果解析成功：
  - 右侧立即更新结果。
- 如果解析失败：
  - 右侧不输出错误格式结果，改为展示错误提示区。
  - 不应清空用户已输入内容。

### 5.3 视图切换

- 切换“文本 / 结构”时，保留当前解析结果，不重新解析输入。
- 视图切换应有平滑过渡，避免页面跳动。

### 5.4 操作反馈

按钮操作需要有即时反馈：

- `格式化`：立即以格式化文本刷新结果区
- `压缩`：立即以压缩文本刷新结果区
- `复制`：复制成功后提示“已复制”
- `下载`：触发浏览器下载 `.json` 文件

### 5.5 移动端交互

当屏幕宽度较小时：

- 左右分栏自动降级为上下布局
- 输入区在上，结果区在下
- 操作按钮排列应保证可点击、不拥挤

## 六、非功能需求

### 6.1 性能

- 正常长度 JSON（几 KB 到几十 KB）应保证即时反馈。
- 对于超大 JSON，需要有保护机制：
  - 当解析或渲染可能造成明显卡顿时，应给出提示。
  - 允许截断展示、提示文本过长，或建议用户压缩输入。

### 6.2 一致性

- 页面视觉必须延续博客现有设计系统：
  - 奶油底
  - amber 强调色
  - 柔和卡片边框
  - 现有字体层级
- 亮色 / 暗色模式都需要适配。

### 6.3 可用性

- 工具应上手即用，无需说明文档。
- 核心操作不超过两步。
- 主要操作应支持点击完成，不依赖快捷键作为必选项。

### 6.4 数据安全

- 所有解析逻辑在浏览器本地完成。
- 不将用户输入发送到服务端。
- 不在本地持久化用户 JSON 内容（当前页面会话内可保留，刷新后清除即可）。

## 七、视觉设计要求

### 7.1 整体风格

延续博客现有“温暖书卷风”设计，不做独立工具站风格。

### 7.2 区块划分

- 输入区与结果区使用一致的卡片容器样式。
- 边框、阴影、圆角遵循站点现有 token。
- 错误提示区应足够醒目，但不应破坏整体温和风格。

### 7.3 文本视图

- JSON key、string、number、boolean、null 应有语法高亮。
- 行号列与内容列需区分层级，避免视觉拥挤。
- 滚动时应保持内容区域稳定，不导致页面抖动。

### 7.4 结构视图

- 树形缩进应统一。
- 折叠 / 展开图标需直观。
- 键名与值之间应有清晰区分。
- 嵌套层级较深时，仍需保持可读性。

---

## 八、边界与异常处理

### 8.1 输入为空

- 不报错。
- 右侧显示占位提示。

### 8.2 输入非法 JSON

- 展示错误提示。
- 保留用户原始输入。
- 不输出格式化结果。

### 8.3 超大 JSON

- 给出性能提示。
- 允许进入受限展示模式或建议用户压缩输入。

### 8.4 特殊字符与转义

- JSON 中的转义字符需正确解析。
- Unicode 内容需正确显示。

### 8.5 非 JSON 文本

- 若输入明显不是 JSON，应提示输入内容无法解析，而不是抛出晦涩错误。

## 九、验收标准

### AC1. 基本格式化

给定一段合法压缩 JSON，点击“格式化”后，右侧文本视图应展示缩进结果，且内容正确。

### AC2. 压缩

给定一段格式化 JSON，点击“压缩”后，右侧文本视图应展示单行 JSON，且内容正确。

### AC3. 结构视图

给定一段合法 JSON，切换到“结构”视图后：

- 对象、数组层级正确
- 支持折叠 / 展开
- 键值显示完整

### AC4. 错误提示

给定一段非法 JSON，页面应展示错误提示，且不覆盖用户输入。

### AC5. 复制与下载

- 点击“复制结果”后，剪贴板内容应与右侧当前结果一致。
- 点击“下载”后，浏览器应可下载 `.json` 文件，文件内容正确。

### AC6. 移动端可用

在窄屏设备或缩小浏览器窗口时，布局自动从左右分栏切换为上下布局，且功能仍完整可用。
